## 2. 核心 Bug 修复清单

### 2.1 专题管理主页面 (HomepageTopicManagementPage)
此页面是专题 CRUD 的核心，也是问题最集中的区域。

| 编号 | 类型 | 严重度 | 修复前描述 | 修复方案 |
| :--- | :--- | :--- | :--- | :--- |
| **BUG-01** | 业务逻辑 | 高 | 删除专题时，未检查是否有关联的活跃投放，可能导致前台首页卡片突然消失。 | 在删除前查询 `topic_placements`，若存在关联投放，则阻断删除并给出明确的错误提示。 |
| **BUG-02** | 业务逻辑 | 中 | 删除专题时，未提示用户将同时删除关联的商品。 | 增加删除前关联商品数量的查询与二次确认弹窗。 |
| **BUG-03** | 数据校验 | 中 | `slug` 字段在数据库中有 UNIQUE 约束，但前端保存前未校验。若重复会导致不友好的数据库报错。 | 在 `handleSave` 中加入异步查库校验，若 `slug` 已存在（排除自身）则阻断保存并友好提示。 |
| **BUG-04** | 业务逻辑 | 高 | 已发布（`published`）的专题允许随意修改 `slug`，这会导致前台已被用户分享或收藏的链接失效。 | 编辑 `published` 状态的专题时，若修改 `slug`，会在输入框下方显示显眼的黄色警告。 |
| **BUG-06** | 功能缺失 | 高 | 正文块（Story Block）编辑器仅支持文本输入，无法指定前台支持的 `heading`、`callout` 等块类型。 | 为每个正文块添加 `block_type` 下拉选择器，支持 `paragraph`、`heading`、`callout` 等类型。 |
| **BUG-07** | 数据一致性 | 中 | 新增正文块时，`block_key` 使用 `Date.now()` 生成。在快速连续点击时可能产生重复 Key，导致 React 渲染异常。 | 引入 `crypto.randomUUID()` 生成绝对唯一的 `block_key`。 |
| **BUG-08** | 数据校验 | 中 | 缺乏时间范围校验，允许 `end_time` 早于 `start_time`，导致专题永远无法在前台展示。 | 在表单 onChange 和提交时增加严格的 `end_time > start_time` 校验，并提供实时 UI 提示。 |
| **BUG-16** | 业务逻辑 | 高 | 将专题状态从 `draft` 变更为 `published` 或 `ready` 时，未校验必填字段（如标题、slug）。 | 状态变更前，强制校验 `slug` 不能为空、且至少包含一种语言的标题。 |
| **BUG-25** | 性能/功能 | 中 | 专题列表查询未分页，随着数据量增加会导致后台加载缓慢和内存溢出。 | 引入基于 `limit` 和 `offset` 的分页机制，每页加载 20 条数据，并增加翻页控件。 |

### 2.2 专题投放管理页面 (TopicPlacementManagementPage)
此页面控制专题在首页 Feed 流中的展示，直接影响 C 端用户。

| 编号 | 类型 | 严重度 | 修复前描述 | 修复方案 |
| :--- | :--- | :--- | :--- | :--- |
| **BUG-09** | 数据校验 | 中 | `feed_position` 虽有 `min={1}` 属性，但手动输入 0 或负数仍可提交，导致前台混排算法异常。 | 在表单提交时，使用 `Number.isInteger` 强制校验值为 1-100 的正整数，并提供实时错误提示。 |
| **BUG-10** | 业务逻辑 | 高 | 允许管理员将 `draft` 或 `offline` 状态的专题关联到投放位。但前台接口只查询 `published` 专题，导致管理员认为系统出现故障。 | 选择非 `published` 专题时，在下拉框下方显示红色警告；提交时增加二次确认弹窗，明确告知前台不可见。 |
| **BUG-11** | 功能缺失 | 中 | 投放表单仅支持上传"默认封面"，但数据库支持 `zh`/`ru`/`tg` 多语言封面。 | 扩展表单 UI，增加中文、俄语、塔吉克语三个独立的封面图片上传组件。 |
| **BUG-12** | 功能缺失 | 低 | 数据库支持覆盖专题的副标题（`subtitle_i18n`），但表单中缺少编辑入口。 | 在"覆盖标题"下方增加"覆盖副标题"的输入区域。 |
| **BUG-20** | 交互体验 | 低 | 投放列表无法直观看出关联专题的当前状态，排查不展示问题时很困难。 | 在投放列表的"关联专题"列，增加醒目的状态徽章（如绿色的 `published`，灰色的 `draft`）。 |

### 2.3 AI 专题生成页面 (AITopicGenerationPage)
此页面负责与大模型交互并生成专题草稿。

| 编号 | 类型 | 严重度 | 修复前描述 | 修复方案 |
| :--- | :--- | :--- | :--- | :--- |
| **BUG-12** | 数据一致性 | 高 | 保存草稿时，`slug` 生成使用 `Date.now().toString(36)`，在高并发或快速保存时极易产生碰撞。 | 改进 `slug` 生成算法，在基础名称后拼接 `crypto.randomUUID().slice(0, 8)`，确保全局唯一。 |
| **BUG-13** | 数据一致性 | 高 | 保存草稿时，未保证 AI 返回的 `story_blocks` 包含有效的 `block_type`，可能导致前台渲染崩溃。 | 在映射数据时，校验并强制赋予合法的 `block_type`（默认回退为 `paragraph`）。 |
| **BUG-24** | React状态 | 中 | `TopicResultPreview` 组件初始化本地状态时使用浅拷贝 `{ ...result }`，导致编辑嵌套对象（如 i18n 文本）时污染了原始数据。 | 改用深拷贝 `JSON.parse(JSON.stringify(result))` 初始化可编辑状态。 |

此组件用于在专题中挂载商品。

| 编号 | 类型 | 严重度 | 修复前描述 | 修复方案 |
| :--- | :--- | :--- | :--- | :--- |
| **BUG-14** | 安全/数据 | 中 | 搜索关键词直接拼接到 PostgREST 查询字符串中。若包含 `.`、`,` 或 `%` 等特殊字符，会导致接口报 400 错误。 | 在构建 `orFilters` 前，使用正则 `replace(/[.,()%_]/g, '')` 剥离可能破坏语法的特殊字符。 |

## 3. 架构与最佳实践建议

在修复上述 Bug 的同时，我们对代码结构进行了梳理，并提出以下长期维护建议：

1. **统一的表单校验层**：当前表单校验（如时间范围、必填项）散落在各个 `handleSubmit` 函数中。建议未来引入 `Zod` 或 `Yup` 等 Schema 校验库，与 `react-hook-form` 结合，提升校验的健壮性和代码可读性。
2. **RLS (Row Level Security) 规范**：本次修复中，已确保所有数据库操作均使用 `adminApi.ts` 中封装的 `adminQuery`、`adminInsert` 等 RPC 代理方法，这正确绕过了 Supabase 的 RLS 限制，符合管理后台的权限设计。请在后续开发中严格保持这一规范。
3. **防抖与节流**：在 `ProductPickerPanel` 中，搜索输入虽然有状态控制，但未实现严格的防抖（Debounce）。建议引入 `useDebounce` Hook，减少用户连续输入时对数据库的无效查询压力。
